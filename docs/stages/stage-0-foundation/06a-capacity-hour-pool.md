# 06a 产能工时池模型

## 问题背景

同一产线同一天可能排多个不同产品的 MO，而不同产品的 UPPH 不同。

如果使用**数量池**（即提前算好"这条线今天能产 X 件"），那么 X 是基于哪个产品的 UPPH 算的？换一个产品就不对了。

::: danger 错误做法
提前计算"产线-日期-可用数量池"，然后逐个 MO 从池中扣减数量。

这在多 UPPH 混排场景下会产生严重偏差。
:::

---

## 正确做法：工时池模型

### 核心思想

> **产线的产能瓶颈是时间，不是数量。**
>
> 每条产线每一天有一个固定的**可用工时**（小时），排产时逐个 MO 消耗工时，而不是消耗"件数"。

### 产能公式

```text
产线日可用工时 = defaultWorkHours（来自日历）

某 MO 在该产线需要的工时 = MO 数量 / UPPH（该产品在该产线的 UPPH）

剩余可用工时 = 当前可用工时 - 已消耗工时
```

### 模型对比

| 维度 | ❌ 数量池模型 | ✅ 工时池模型 |
| --- | --- | --- |
| 池的单位 | PCS（件） | 小时（h） |
| 池的计算 | 日可产数量 = UPPH × 工时 | 日可用工时 = defaultWorkHours |
| 扣减方式 | 扣减 MO 数量 | 扣减 MO 耗时 = Qty / UPPH |
| 多 UPPH 混排 | ❌ UPPH 不同时无法混合扣减 | ✅ 不同 UPPH 的 MO 各自换算工时后扣减 |
| 适用场景 | 仅单产品单线单日 | 单产品或多产品混排均可 |

## 示例演算

### 场景

- 产线 3F3，当天可用工时 = **10 小时**
- MO-A：产品 ITEM001，数量 600，UPPH = 100
- MO-B：产品 ITEM002，数量 400，UPPH = 80

### ❌ 数量池做法（错误）

```text
假设用 ITEM001 的 UPPH 预算：
  日可产数量 = 100 × 10 = 1000

排 MO-A：剩余 1000 - 600 = 400
排 MO-B：剩余 400 - 400 = 0 ← 看起来刚好排完

但实际上 MO-B 的 UPPH 是 80，不是 100！
MO-B 实际需要 400 / 80 = 5h，不是 400 / 100 = 4h。
总耗时 = 6h + 5h = 11h > 10h ← 实际超产能！
```

### ✅ 工时池做法（正确）

```text
初始工时池 = 10h

排 MO-A：
  耗时 = 600 / 100 = 6h
  剩余工时 = 10 - 6 = 4h

排 MO-B：
  耗时 = 400 / 80 = 5h
  剩余工时 = 4 - 5 = -1h ← 不够！

结论：MO-B 当天只能排 4h × 80 = 320 件
      剩余 80 件需要排到下一个工作日
```

## 工时池状态模型

排产过程中，每条产线每个日期维护一个**工时池状态**：

```text
line_day_pool = {
  lineCode:       "3F3",
  date:           "2026-05-13",
  totalHours:     10.0,        // 来自日历
  consumedHours:  0.0,         // 已占用工时
  remainingHours: 10.0,        // 剩余工时
  allocations: [               // 占用明细
    // { mo, fgItemCode, upph, qty, hours }
  ]
}
```

每排入一个 MO 时，更新工时池：

```text
1. 计算 MO 耗时 = min(remainingQty, remainingHours × UPPH) 对应的工时
2. actualQty = min(remainingQty, remainingHours × UPPH)
3. actualHours = actualQty / UPPH
4. consumedHours += actualHours
5. remainingHours -= actualHours
6. 记录 allocation
7. 如果 remainingQty > actualQty，则需要继续排到下一日
```

## 对阶段 0 的要求

工时池模型要求阶段 0 必须确认以下事项：

| 确认项 | 说明 | 影响 |
| --- | --- | --- |
| UPPH 必须精确到 **产线 × 产品 × 工段** | 不能只有产线级 UPPH，否则不同产品无法正确换算工时 | 工时计算准确性 |
| UPPH 必须是 **线体级** | 人均 UPPH 需要额外乘以人数，增加复杂度和错误风险 | 公式简洁性 |
| 日历工时必须是 **可用工时**（非自然小时） | 10h 是实际可生产时间，不含休息、交接班等 | 工时池准确性 |
| 不同产品的换线时间是否扣减工时 | 如果换线需要 0.5h，应从工时池中扣除 | 阶段 6 引入，阶段 0 先记录 |

::: warning 阶段 0 必须确认
- UPPH 的粒度是否足够精细（产线 × 产品 × 工段）？
- UPPH 值是否代表"线体每小时净产出"？
- 日历中的 defaultWorkHours 是否代表"可用于生产的净工时"？

如果以上三项无法确认，工时池模型的结果将不可信。
:::

## 异常场景

| 异常 | 触发条件 | 级别 | 处理方式 |
| --- | --- | --- | --- |
| HOUR_POOL_EXHAUSTED | 当天工时池已耗尽，仍有 MO 待排 | INFO | 自动溢出到下一工作日 |
| PARTIAL_DAY_ALLOCATION | MO 数量无法在当天排完，部分溢出 | INFO | 记录当日实排数量与溢出数量 |
| UPPH_GRANULARITY_MISMATCH | 找到的 UPPH 不是产线×产品级，而是更粗粒度 | WARNING | 使用粗粒度值但标记不精确 |

## 与 06 产能与日历口径的关系

[06 产能与日历口径](./06-capacity-calendar-baseline) 定义了基础公式 `日可产数量 = UPPH × 工时`。

本页在此基础上进一步明确：**当多个 MO 共享同一产线同一天时，产能的共享单位是工时（小时），不是数量（件）。**

```text
06 定义的公式：日可产数量 = UPPH × 工时
  ↓
06a 补充的模型：产线日产能 = 一个工时池
                每个 MO 按自己的 UPPH 消耗工时
                池耗尽则溢出到下一日
```
