# 06a 产能工时池模型

## 问题背景

同一产线同一天可能排多个不同产品的 MO，而不同产品的 UPPH 不同。

如果使用**数量池**（即提前算好"这条线今天能产 X 件"），那么 X 是基于哪个产品的 UPPH 算的？换一个产品就不对了。

::: danger 错误做法
提前计算"产线-日期-可用数量池"，然后逐个 MO 从池中扣减数量。

这在多 UPPH 混排场景下会产生严重偏差。
:::

---

## 正确做法：工时池模型

### 核心思想

> **产线的产能瓶颈是时间，不是数量。**
>
> 每条产线每一天有一个固定的**可用工时**（小时），排产时逐个 MO 消耗工时，而不是消耗"件数"。

### 产能公式

UPPH 已确认为**人均每小时产出**，因此需要引入人数：

```text
产线日可用工时 = defaultWorkHours（来自日历）

产线日线体产出率 = UPPH × Headcount（人数）

某 MO 在该产线需要的工时 = MO 数量 / (UPPH × Headcount)

剩余可用工时 = 当前可用工时 - 已消耗工时
```

### 模型对比

| 维度 | ❌ 数量池模型 | ✅ 工时池模型 |
| --- | --- | --- |
| 池的单位 | PCS（件） | 小时（h） |
| 池的计算 | 日可产数量 = UPPH × 人数 × 工时 | 日可用工时 = defaultWorkHours |
| 扣减方式 | 扣减 MO 数量 | 扣减 MO 耗时 = Qty / (UPPH × Headcount) |
| 多 UPPH 混排 | ❌ UPPH 不同时无法混合扣减 | ✅ 不同 UPPH 的 MO 各自换算工时后扣减 |
| 适用场景 | 仅单产品单线单日 | 单产品或多产品混排均可 |

## 示例演算

### 场景

- 产线 3F3，当天可用工时 = **10 小时**，排班人数 = **5 人**
- MO-A：产品 ITEM001，数量 600，UPPH = 100（人均），线体产出率 = 100 × 5 = 500/h
- MO-B：产品 ITEM002，数量 400，UPPH = 80（人均），线体产出率 = 80 × 5 = 400/h

### ❌ 数量池做法（错误）

```text
假设用 ITEM001 的 UPPH 预算（人均100 × 5人）：
  日可产数量 = 100 × 5 × 10 = 5000

排 MO-A：剩余 5000 - 600 = 4400
排 MO-B：剩余 4400 - 400 = 4000 ← 看起来绰绰有余

但如果换了一个 UPPH 差异更大的产品，偏差就会累积。
数量池模型的根本问题在于：不同产品的 UPPH 不同，
预算时用的 UPPH 和实际排入 MO 的 UPPH 不一致。
```

### ✅ 工时池做法（正确）

```text
初始工时池 = 10h，排班人数 = 5 人

排 MO-A（UPPH=100，人均）：
  线体产出率 = 100 × 5 = 500/h
  耗时 = 600 / 500 = 1.2h
  剩余工时 = 10 - 1.2 = 8.8h

排 MO-B（UPPH=80，人均）：
  线体产出率 = 80 × 5 = 400/h
  耗时 = 400 / 400 = 1.0h
  剩余工时 = 8.8 - 1.0 = 7.8h

结论：两个 MO 都能在当天排完，剩余 7.8h 可用工时
```

## 工时池状态模型

排产过程中，每条产线每个日期维护一个**工时池状态**：

```text
line_day_pool = {
  lineCode:       "3F3",
  date:           "2026-05-13",
  headcount:      5,           // 排班人数
  totalHours:     10.0,        // 来自日历
  consumedHours:  0.0,         // 已占用工时
  remainingHours: 10.0,        // 剩余工时
  allocations: [               // 占用明细
    // { mo, itemId, upph, headcount, lineRate, qty, hours }
  ]
}
```

每排入一个 MO 时，更新工时池：

```text
1. 计算线体产出率 lineRate = UPPH × Headcount
2. actualQty = min(remainingQty, remainingHours × lineRate)
3. actualHours = actualQty / lineRate
4. consumedHours += actualHours
5. remainingHours -= actualHours
6. 记录 allocation
7. 如果 remainingQty > actualQty，则需要继续排到下一日
```

## 对阶段 0 的要求

工时池模型要求阶段 0 必须确认以下事项：

| 确认项 | 说明 | 状态 |
| --- | --- | --- |
| UPPH 必须精确到 **产线 × 产品 × 工段** | 不能只有产线级 UPPH，否则不同产品无法正确换算工时 | 待确认粒度 |
| UPPH 是 **人均每小时产出** | 已确认，公式需要乘以 Headcount（排班人数） | ✅ 已确认 |
| 需要维护 **Headcount（排班人数）** | 人均 UPPH 需要额外乘以排班人数才能得到线体产出率 | 需确认数据来源 |
| 日历工时必须是 **可用工时**（非自然小时） | 10h 是实际可生产时间，不含休息、交接班等 | 工时池准确性 |
| 不同产品的换线时间是否扣减工时 | 如果换线需要 0.5h，应从工时池中扣除 | 阶段 6 引入，阶段 0 先记录 |

::: warning 阶段 0 仍需确认
- UPPH 的粒度是否足够精细（产线 × 产品 × 工段）？
- 排班人数（Headcount）的数据来源是什么？是固定值还是每日变动？
- 日历中的 defaultWorkHours 是否代表"可用于生产的净工时"？

如果以上三项无法确认，工时池模型的结果将不可信。
:::

## 异常场景

| 异常 | 触发条件 | 级别 | 处理方式 |
| --- | --- | --- | --- |
| HOUR_POOL_EXHAUSTED | 当天工时池已耗尽，仍有 MO 待排 | INFO | 自动溢出到下一工作日 |
| PARTIAL_DAY_ALLOCATION | MO 数量无法在当天排完，部分溢出 | INFO | 记录当日实排数量与溢出数量 |
| UPPH_GRANULARITY_MISMATCH | 找到的 UPPH 不是产线×产品级，而是更粗粒度 | WARNING | 使用粗粒度值但标记不精确 |

## 与 06 产能与日历口径的关系

[06 产能与日历口径](./06-capacity-calendar-baseline) 定义了基础公式 `日可产数量 = UPPH × 人数 × 工时`。

本页在此基础上进一步明确：**当多个 MO 共享同一产线同一天时，产能的共享单位是工时（小时），不是数量（件）。**

```text
06 定义的公式：日可产数量 = UPPH × Headcount × 工时
  ↓
06a 补充的模型：产线日产能 = 一个工时池
                线体产出率 = UPPH × Headcount
                每个 MO 按自己的线体产出率消耗工时
                池耗尽则溢出到下一日
```
