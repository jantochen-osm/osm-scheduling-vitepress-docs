# 风险与控制措施

## 已识别风险清单

阶段 0 的主要风险集中在**数据口径不清**和**业务边界不明**两类，下表列出已识别风险及应对措施。

---

### 风险一：AX 字段口径不一致

| 项目 | 内容 |
| --- | --- |
| 风险描述 | 同一业务概念在 AX 2012 中可能存在多个字段，或不同版本的 AX 字段名不同，导致数据提取错误。 |
| 影响 | 订单筛选条件错误、UPPH 匹配失败、排产结果偏差 |
| 发生概率 | 高 |
| 控制措施 | 建立**字段映射表**，记录 AX 字段名 → OSM 逻辑字段名的对应关系，并由 IT 和业务团队双签确认 |
| 应对方案 | 在中间库建立标准化视图（View），屏蔽 AX 原始字段差异 |
| 责任人 | IT 团队 |

---

### 风险二：UPPH 口径不清

| 项目 | 内容 |
| --- | --- |
| 风险描述 | 工程部提供的 UPPH 可能混用"线体 UPPH"和"人均 UPPH"，导致产能计算严重偏差。 |
| 影响 | 排产结果产能计算错误，导致大量逾期或排产数量不准确 |
| 发生概率 | 高 |
| 控制措施 | 阶段 0 必须**明确口径**：统一使用线体 UPPH，并在字段清单中注明换算关系（如有人均 UPPH 数据，需补充人员数量字段）|
| 应对方案 | 如无法确认，使用两个字段分别记录（`upph_line` 和 `upph_per_person`），引擎中选择线体口径 |
| 责任人 | 工程团队 + 系统设计团队 |

---

### 风险三：OSM_Category 字段来源不明

| 项目 | 内容 |
| --- | --- |
| 风险描述 | AX 中可能没有直接的 EE/ESG 分类字段，需要通过产品属性、Pool、产线归属等多条件推导。 |
| 影响 | 无法正确将订单路由到 EE 或 ESG 产线规则，导致批量分类异常 |
| 发生概率 | 中 |
| 控制措施 | 在阶段 0 确认 OSM_Category 的派生规则，并形成规则文档 |
| 应对方案 | 在中间库或 OSM 系统建立 Category 映射表，支持手动维护和规则自动推导两种方式 |
| 责任人 | 业务团队 + IT 团队 |

---

### 风险四：样例数据覆盖不足

| 项目 | 内容 |
| --- | --- |
| 风险描述 | 收集的样例订单太少或不具代表性，导致阶段 1 开发时遇到未预料的数据结构问题。 |
| 影响 | 阶段 1 开发中途返工，延误进度 |
| 发生概率 | 中 |
| 控制措施 | 样例数据集必须覆盖以下场景：EE + ESG 两类、有路线 + 无路线（测试异常）、有 UPPH + 无 UPPH（测试异常）、不同 DlvDate 跨度 |
| 应对方案 | 如无法获取真实数据，由业务团队提供脱敏或模拟数据 |
| 责任人 | 业务团队 + 数据团队 |

---

### 风险五：业务边界争议

| 项目 | 内容 |
| --- | --- |
| 风险描述 | 不同业务方对"OSM 排产系统排什么"的理解不一致，导致范围频繁变动。 |
| 影响 | 需求蔓延，阶段 0 拖延，影响整体建设节奏 |
| 发生概率 | 中 |
| 控制措施 | 阶段 0 的业务边界确认文档（D0-01）必须经过**正式评审并签字**，作为基准 |
| 应对方案 | 变更范围需走变更流程，评估对后续阶段的影响后再执行 |
| 责任人 | 项目负责人 |

---

### 风险六：4F7/4F8 产线定位未确认

| 项目 | 内容 |
| --- | --- |
| 风险描述 | 4F7、4F8 产线的业务定位（是否专线、产品归属）尚未确认，可能影响 ESG 场景的完整性。 |
| 影响 | 阶段 3 ESG 扩展时需要返工 |
| 发生概率 | 低（短期影响低，长期需关注） |
| 控制措施 | 阶段 0 记录为"待确认"，在阶段 3 开始前完成确认 |
| 应对方案 | 作为 `reference/tbd` 中的待确认事项跟踪 |
| 责任人 | 业务团队 |

---
